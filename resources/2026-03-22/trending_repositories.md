## langchain-ai/deepagents — 「とりあえず動くエージェント」が5行で手に入るLangChain公式フレームワーク

**GitHub**: https://github.com/langchain-ai/deepagents
**ドキュメント**: https://github.com/langchain-ai/deepagents#readme

### どんなプロジェクトか

AIエージェントを自分で組み立てようとすると、意外なところで時間を取られる。ツールの定義、コンテキストウィンドウの管理、サブエージェントの起動と結果の回収、長い会話の要約処理。フレームワークを選んでも、こうした「配管工事」に半日費やしてから、やっと本題のロジックに取り掛かれる、という経験は珍しくない。

Deep Agentsは、LangChainチームがこの問題に対して出した回答だ。「batteries-included（全部入り）」を掲げたエージェントフレームワークで、計画立案、ファイル操作、シェル実行、サブエージェント委任、コンテキスト自動管理という5つの基盤機能が最初から組み込まれている。`pip install deepagents`してから、動くエージェントを手に入れるまでに必要なコードは5行だ。

### 何が従来のLangChainエージェントと違うのか

LangChainには以前からエージェント構築の仕組みがあったが、柔軟性を重視した設計ゆえに、開発者が自分でツールを選び、メモリを設定し、実行ループを組み立てる必要があった。Deep Agentsは逆のアプローチを取っている。「ほとんどのエージェントが必要とする機能」をデフォルトで提供し、不要なものだけ外す設計だ。

中核にあるのは`write_todos`による計画機能と、`task`によるサブエージェント委任だ。複雑なタスクを受け取ると、エージェントはまずTodoリストに分解し、各タスクを独立したコンテキストウィンドウを持つサブエージェントに振り分ける。親エージェントのコンテキストが膨らまないため、長時間の複雑なタスクでも安定して動作する。

コンテキスト管理も自動化されている。会話が長くなると自動的に要約が挿入され、大きな出力はファイルシステムにオフロードされる。開発者が明示的にメモリ管理のコードを書く必要がない。

技術的には、内部でLangGraphのコンパイル済みグラフを返す設計になっている。つまり、LangGraphが持つストリーミング、チェックポイント、永続化、Studio連携といったプロダクション向け機能がそのまま使える。フレームワークの学習コストを下げつつ、スケールする道を最初から確保しているわけだ。

### CLIとMCPで広がる実用性

ライブラリとしてPythonから使うだけでなく、CLI版も提供されている。こちらにはWeb検索、リモートサンドボックス、永続メモリ、人間の承認を挟むhuman-in-the-loopワークフローが追加されており、ターミナルからすぐに使い始められる。

MCP（Model Context Protocol）にも`langchain-mcp-adapters`経由で対応しているため、既存のMCPツールエコシステムとの接続も柔軟だ。

### なぜ今、これが使われているのか

エージェントフレームワークは飽和状態に近づいているが、「すぐ動く」と「本番で使える」を両立しているものは意外と少ない。軽量なフレームワークは本番投入時にスケーラビリティの壁にぶつかり、重厚なフレームワークは学習コストが高くてプロトタイピングに向かない。

Deep Agentsは、LangGraphというプロダクション実績のあるランタイムの上に、opinions（設計方針の決め打ち）を乗せることで、この二者択一を解消している。101人のコントリビューターと74回のリリースが示す通り、開発ペースは速く、すでにv0.4.12に到達している。

「とりあえず試す」から「本番に載せる」までの距離が短いフレームワークを探しているなら、最初に検討する価値がある。MITライセンス。

## andrewyng/context-hub — コーディングエージェントの「物忘れ」を根本から解決するドキュメンテーション基盤

**GitHub**: https://github.com/andrewyng/context-hub

### どんなプロジェクトか

コーディングエージェントを日常的に使っていると、繰り返し遭遇する問題がある。APIの幻覚（ハルシネーション）だ。存在しないメソッドを自信満々に呼び出したり、廃止されたパラメータを使ったりする。公式ドキュメントを参照させても、バージョンが古かったり、エージェントが必要な箇所を見つけられなかったりする。

Context Hubは、この「エージェントが正確なAPIドキュメントにアクセスできない」という問題を、コミュニティ駆動のキュレーション済みドキュメント基盤で解決するプロジェクトだ。CLIツール`chub`を通じて、エージェントがバージョン管理された正確なドキュメントを取得し、しかも使うたびに賢くなっていく仕組みを提供している。

### 「使うほど賢くなる」フィードバックループ

Context Hubの設計で最も本質的なのは、単なるドキュメント配信ではなく、フィードバックループが組み込まれている点だ。

エージェントがドキュメントを取得して作業する中で、記載されていないワークアラウンドや、ドキュメントのギャップを発見することがある。Context Hubでは、こうした発見を「アノテーション」としてローカルに保存できる。保存されたアノテーションは次回以降のセッションで自動的にドキュメントに付与されるため、同じ問題に二度遭遇しない。

さらに、ドキュメントの品質に対するフィードバック（良い/悪い）をコンテンツのメンテナーに送る仕組みもある。個々のエージェントの学びがコミュニティ全体のドキュメント品質向上につながる、という好循環が設計に織り込まれている。

### トークン効率を考えたインクリメンタル取得

もう一つ実用的なのが、ドキュメントの段階的な取得だ。ドキュメントはメインのエントリポイントと複数の参照ファイルで構成されているが、エージェントは必要な部分だけを取得できる。全文を一気にコンテキストに詰め込む必要がないため、トークン消費を抑えられる。

言語別のバリアントにも対応しており、`chub get openai/chat --lang py`のようにPython向けのドキュメントだけを取得することもできる。

### なぜ今、これが必要とされているのか

コーディングエージェントの能力が上がるにつれて、「エージェントが使うドキュメントの品質」がボトルネックになるケースが増えている。モデル自体の推論能力は十分に高いのに、参照する情報が不正確だから間違える。RAGで公式ドキュメントを検索させても、チャンキングの問題で必要な情報が欠落したり、古いバージョンの情報が混入したりする。

Context Hubは、この問題をモデル側やRAG側ではなく、「ドキュメントそのもの」の層で解決しようとしている。Markdownベースのキュレーション済みコンテンツをYAMLフロントマターで構造化し、バージョン管理する。人間のメンテナーとエージェントのフィードバックの両方でドキュメントが改善されていくため、時間が経つほど品質が上がる。

npmパッケージ`@aisuite/chub`として配布されており、導入は`npm install -g @aisuite/chub`で完了する。MITライセンスで公開されていて、ドキュメントのコントリビューションも歓迎されている。APIプロバイダーやフレームワーク開発者が自分のプロダクトのドキュメントをContext Hubに提供すれば、エージェント経由での利用体験が直接改善される。

エージェントの「知識の質」を底上げするインフラとして、開発者もライブラリ作者も注目しておくべきプロジェクトだ。
